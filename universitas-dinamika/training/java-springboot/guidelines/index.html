@RestController

        return ResponseEntity.ok(productService.findAll());

        return ResponseEntity.ok(productService.findById(id));

    public ResponseEntity<ProductResponse> create(
            @Valid @RequestBody CreateProductRequest request) {
        ProductResponse created = productService.create(request);
        return ResponseEntity.status(HttpStatus.CREATED).body(created);
    }

    @PutMapping("/{id}")
    public ResponseEntity<ProductResponse> update(
            @PathVariable Long id,

        return ResponseEntity.ok(productService.update(id, request));
    }

    @DeleteMapping("/{id}")
    public ResponseEntity<Void> delete(@PathVariable Long id) {
        productService.delete(id);
        return ResponseEntity.noContent().build();

API Versioning

API yang sudah dipakai client tidak bisa diubah sembarangan. Versioning memungkinkan kamu merilis breaking change di v2 tanpa merusak client yang masih pakai v1.

// Tujuan Lab
  • Memahami tiga strategi versioning: URL, header, dan request param
  • Implementasi URL versioning (paling umum)
  • Strategi deprecation yang baik
  • Memilih strategi yang tepat sesuai konteks

① Perbandingan Strategi Versioning

StrategiContoh RequestKelebihanKekurangan
URL Path GET /api/v1/products
GET /api/v2/products		 Paling mudah dipahami, mudah di-test di browser/Postman, mudah dicache URL berubah, tidak "REST pure" (versi bukan resource)
Request Param GET /api/products?version=2 URL sama, mudah di-test Tidak standar, bisa tercampur dengan filter param bisnis
Header Accept: application/vnd.myapp.v2+json URL bersih, "REST pure", header eksplisit Tidak bisa dibuka langsung di browser, Postman butuh setup header
Rekomendasi Gunakan URL Path untuk API publik. Gunakan Header jika API internal dan ingin URL tetap bersih.

public class ProductSpecification {

  public static Specification<Product> hasKeyword(String keyword) {
    return (root, query, cb) -> keyword == null ? null :
        cb.like(cb.lower(root.get("name")), "%" + keyword.toLowerCase() + "%");
  }

  public static Specification<Product> inCategory(Long categoryId) {
    return (root, query, cb) -> categoryId == null ? null :
        cb.equal(root.get("category").get("id"), categoryId);
  }

  public static Specification<Product> priceBetween(BigDecimal min, BigDecimal max) {
    return (root, query, cb) -> {
      if (min == null && max == null) return null;
      if (min == null) return cb.lessThanOrEqualTo(root.get("price"), max);
      if (max == null) return cb.greaterThanOrEqualTo(root.get("price"), min);
      return cb.between(root.get("price"), min, max);
    };
  }
}

① @Transactional Propagation

PropagationPerilakuKapan digunakan
REQUIRED (default)Gabung transaksi yang ada. Jika belum ada, buat baru.Hampir semua service method
REQUIRES_NEWSelalu buat transaksi baru. Suspend transaksi yang ada.Audit log — harus tersimpan meski transaksi utama rollback
NESTEDBuat savepoint dalam transaksi yang ada. Bisa partial rollback.Loop yang sebagian boleh gagal tanpa rollback semua

NEVERThrow exception jika ada transaksi aktif.Operasi yang harus dijamin tidak transaksional
MANDATORYThrow exception jika TIDAK ada transaksi aktif.Method yang hanya boleh dipanggil dalam konteks transaksi

// ─── Skenario deadlock ───
// Thread A: lock account 1 → tunggu lock account 2
// Thread B: lock account 2 → tunggu lock account 1  ← DEADLOCK

// ─── Solusi 1: Consistent lock ordering — selalu lock dengan ID kecil duluan ───
@Transactional
public void transfer(Long fromId, Long toId, BigDecimal amount) {
    Long firstId  = Math.min(fromId, toId);
    Long secondId = Math.max(fromId, toId);

    // semua thread selalu lock dalam urutan ID ascending — tidak bisa deadlock
    Account first  = accountRepo.findByIdForUpdate(firstId);   // SELECT ... FOR UPDATE
    Account second = accountRepo.findByIdForUpdate(secondId);
    // lakukan transfer...
}

@Query("SELECT a FROM Account a WHERE a.id = :id")

    private Long id;
    @Column(nullable = false, unique = true)
    private String token;               // UUID acak, bukan JWT

    private Instant expiresAt;

@Service
@RequiredArgsConstructor
public class AuthService {

    public TokenResponse refresh(String rawToken) {
        RefreshToken token = refreshTokenRepo.findByToken(rawToken)
            .orElseThrow(() -> new InvalidTokenException("Refresh token tidak valid"));

        if (token.isRevoked() || token.getExpiresAt().isBefore(Instant.now())) {
            // token di-reuse setelah revoke = possible theft, revoke semua token user
            refreshTokenRepo.revokeAllByUser(token.getUser());
            throw new InvalidTokenException("Token expired atau sudah dipakai");
        }

        // Rotate: hapus yang lama, buat yang baru
        token.setRevoked(true);
        refreshTokenRepo.save(token);

        String newRefresh = generateAndSave(token.getUser());
        String newAccess  = jwtService.generate(token.getUser());
        return new TokenResponse(newAccess, newRefresh);
    }
}

@Service
@RequiredArgsConstructor
@Slf4j                                   // Lombok: inject log field
public class ProductService {

    private final ProductRepository repo;
    private final ProductMapper mapper;  // MapStruct

    @Transactional(readOnly = true)
    public Page<ProductResponse> findAll(Pageable pageable) {
        return repo.findAll(pageable).map(mapper::toResponse);
    }

    @Transactional
    public ProductResponse create(CreateProductRequest req) {
        log.info("Creating product: {}", req.getName());

        if (repo.findByName(req.getName()).isPresent()) {
            throw new DuplicateResourceException("Product sudah ada: " + req.getName());
        }

        Product product = mapper.toEntity(req);
        Product saved = repo.save(product);
        log.info("Product created with id: {}", saved.getId());

        return mapper.toResponse(saved);
    }
}

@Async — Non-blocking Background Task

Operasi lambat (kirim email, generate laporan) sebaiknya tidak memblokir thread HTTP. Dengan @Async, method dieksekusi di thread pool terpisah. Penting: method @Async harus di kelas yang berbeda dengan pemanggilnya (proxy Spring tidak bisa intercept self-call).

// ─── Konfigurasi Thread Pool ───
@Configuration
@EnableAsync
public class AsyncConfig {

    @Bean("emailTaskExecutor")
    public TaskExecutor emailTaskExecutor() {
        ThreadPoolTaskExecutor exec = new ThreadPoolTaskExecutor();
        exec.setCorePoolSize(2);
        exec.setMaxPoolSize(5);
        exec.setQueueCapacity(100);
        exec.setThreadNamePrefix("email-");
        exec.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
        return exec;
    }
}

// ─── Service dengan @Async ───
@Service
@Slf4j
public class EmailService {

            Thread.currentThread().getName());

    }
}

CorrelationIdFilter.java — inject correlationId ke setiap request
java
@Component
@Order(Ordered.HIGHEST_PRECEDENCE)
public class CorrelationIdFilter extends OncePerRequestFilter {

    private static final String HEADER = "X-Correlation-Id";

    @Override
    protected void doFilterInternal(HttpServletRequest req, HttpServletResponse res,
                                      FilterChain chain) throws IOException, ServletException {
        String id = Optional.ofNullable(req.getHeader(HEADER))
            .orElseGet(() -> UUID.randomUUID().toString());
        MDC.put("correlationId", id);   // semua log di thread ini punya field ini
        res.setHeader(HEADER, id);        // balikkan ke client untuk debugging
        try {
            chain.doFilter(req, res);
        } finally {
            MDC.clear();                  // wajib clear agar thread pool tidak bocor
        }
    }
}

SELECT p.*, c.name AS category_name
FROM products p

ORDER BY p.created_at DESC

  });

⑦ Circuit Breaker — Resilience4j

Circuit Breaker mencegah cascade failure: jika product-service lambat atau down, order-service tidak ikut hang menunggu. Setelah threshold failure tercapai, circuit "terbuka" dan langsung return fallback — memberikan waktu recovery ke service yang bermasalah.

Resilience4j — Circuit Breaker + Retry + Timeout
java
# application.yml — konfigurasi circuit breaker
resilience4j:
  circuitbreaker:
    instances:
      product-service:
        slidingWindowSize: 10              # evaluasi 10 request terakhir
        failureRateThreshold: 50           # buka circuit jika 50% gagal
        waitDurationInOpenState: 10s       # tunggu 10 detik sebelum coba lagi
        permittedCallsInHalfOpenState: 3   # coba 3 request saat half-open
  retry:
    instances:
      product-service:
        maxAttempts: 3
        waitDuration: 500ms
  timelimiter:
    instances:
      product-service:
        timeoutDuration: 2s              # timeout jika > 2 detik

// ─── Penggunaan di OrderService ───
@CircuitBreaker(name = "product-service", fallbackMethod = "getProductFallback")
@Retry(name = "product-service")
@TimeLimiter(name = "product-service")
public CompletableFuture<ProductResponse> getProduct(Long id) {
    return CompletableFuture.supplyAsync(() -> productClient.getProduct(id));
}

// Fallback dipanggil saat circuit terbuka atau timeout
public CompletableFuture<ProductResponse> getProductFallback(Long id, Throwable t) {
    log.warn("product-service unavailable for id={}: {}", id, t.getMessage());
    return CompletableFuture.completedFuture(ProductResponse.unavailable(id));
}

⑨ Kapan Gunakan Microservice?

⚠️Jangan over-engineer. Microservice menambah kompleksitas: distributed transaction, network latency, service discovery, observability — semua harus dikelola. Mulai dengan monolith yang bersih (layered architecture). Pecah menjadi microservice hanya saat ada bottleneck nyata: satu modul terlalu sering deploy, satu domain butuh scale berbeda, atau tim sudah cukup besar (>8 orang) untuk mengelola service terpisah.
SinyalArtinya
Satu modul sering deploy, yang lain jarangKandidat untuk dipecah ke service tersendiri
Satu domain butuh scale 10× lebih banyakMicroservice memungkinkan scale hanya bagian itu
Tim terbagi per domain (misal: tim order, tim produk)Microservice align dengan struktur tim (Conway's Law)
Database monolith jadi bottleneckDatabase per service menghilangkan shared-schema coupling
Butuh tech stack berbeda per modulMicroservice memungkinkan polyglot technology
