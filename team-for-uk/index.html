Risk & Mitigation

What Happens If the Timeline Slips

Delays happen. Below is how we handle scope creep, blockers, and sprint overruns — so CPDme always knows the plan before a problem becomes a surprise.

🔁

Sprint Overrun — Task Carries Over, Not Dropped

If a task isn't completed within its sprint, it rolls into the next sprint as a priority item — it doesn't quietly disappear. Leni reports the carryover in the weekly progress report with a revised estimate and root cause (e.g. unexpected complexity, dependency block, or scope change).

🔀

Scope Change — Backlog Reprioritisation, Not Silent Additions

If CPDme adds new tasks mid-sprint, we'll flag the trade-off: something of equal weight moves to the next sprint, or we extend the sprint with CPDme's agreement. We never silently absorb new scope without adjusting the timeline.

🚧

External Dependency Blocker — Escalated Within 24 Hours

If progress is blocked by something outside our control — a third-party API outage, missing credentials, or a decision needed from CPDme — we flag it in standup the same day and escalate via Slack/Teams if unresolved within 24 hours. We don't sit on blockers.

📊

Month-End Review — Timeline Reset If Needed

At the end of each month, we run a brief retrospective with CPDme. If we're consistently running behind, we adjust sprint capacity estimates or re-scope the roadmap together — rather than pretending we'll catch up and then missing again.

⚡

Critical Bug in Production — Dropped Into Current Sprint Immediately

Any Sentry-flagged production error or user-reported critical bug gets pulled into the active sprint immediately, displacing a lower-priority backlog item. Production stability takes precedence over scheduled features — CPDme's users are never left broken while we work on new things.

06 — Special Requirements & Dependencies

Known Dependencies & Constraints

🤖

Android 15 (API Level 35) Compliance

Google Play Store now requires all new app submissions to target Android 15 (API 35). This impacts the Flutter mobile codebase — SDK version upgrade, dependency audit, and Play Store re-submission are required. Addressed across Sprint 2–3 by the mobile engineer.

🪟

Microsoft SSO — Global Integration

CPDme's corporate customers expect Microsoft/Azure AD SSO as a login option. This requires OAuth2 app registration in Azure, callback URL configuration, org-claim mapping, and fallback authentication logic. Must include audit logging for all SSO events. Scheduled in Sprint 3.

🇬🇧

UK GDPR — Account Deletion & Data Erasure

Article 17 (Right to Erasure) requires that user deletion requests from both in-app and Intercom channels are fully processed — not just soft-deleted but data anonymised across all systems (database, Intercom, email, Katra). Currently broken for both vectors. Sprint 1 priority.

🔗

Intercom Integration Dependency

Several features depend on Intercom as the customer support and communication layer — account deletion requests, tag syncing, certificate notifications, and user activity events. Any Intercom API changes require coordinated updates across backend and webhook handlers.

🏛️

Governing Body Standards Versioning

GMC, HCPC, PHECC, PSI, AHPRA, CMI, BACP, and others maintain their own CPD standards formats which change periodically. CPDme must track standards versions and map them correctly to user portfolios. Standards changes (e.g. GMC text update) require a coordinated content and data migration — not just a front-end text swap.

⚠️

Sentry Error Monitoring — Production Visibility

Sentry is already capturing production exceptions (e.g. the file download failure in Sprint 1). We require read/write access to the CPDme Sentry project to triage, assign, and resolve production errors as part of normal sprint work. Alerts should be routed to our shared Slack channel.

🌍

Multi-Timezone & Multi-Org User Logic

Users can be members of multiple organisations across different governing bodies (e.g. a physiotherapist who is also a CSP and AHPRA member). Supervision, portfolio generation, and CPD hour counting must correctly scope data per-organisation without cross-contamination. This is a dependency for several Sprint 4 features.

🔑

Access to Existing Codebase & Infrastructure

To work at full velocity from day one, we require: read/write access to the CPDme Git repositories, staging environment credentials, Sentry project access, Intercom workspace access, and an onboarding call with the current technical team to understand deployment pipelines and known system quirks.
